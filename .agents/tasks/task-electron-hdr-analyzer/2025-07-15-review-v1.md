# Electron port of HDR Video Analyzer

A greenfield Electron desktop app that ports the Python `HDRAnalyzer_V8.py` tool to JavaScript. The app shells out to FFmpeg for frame decoding, performs per-pixel PQ EOTF and gamut classification in the main process, and renders a 3-panel chart via `node-canvas`. Eight files, ~1400 lines, zero dependencies installable in this environment so this review covers source correctness only.

**Watch for:** Main-process blocking during frame analysis (confirmed) will freeze the UI for multi-minute videos. The FFmpeg child process is never killed on user cancellation or window close, leaving orphaned processes (confirmed). A subtle gamut-classification divergence from the Python original silently drops pixels with low-confidence chromaticity into Rec.2020 instead of Rec.709 (confirmed).

## High-level view

The color-science math — PQ EOTF, barycentric gamut test, Rec.2020→XYZ conversion — is ported faithfully, with one behavioral difference in how the "confidence mask" (low XYZ-sum pixels) is handled that shifts gamut ratios toward wider gamut compared to the Python version.

Frame analysis runs synchronously on the main thread inside `stdout.on('data')`, performing millions of floating-point operations per frame. For a 2-hour film this means the Electron event loop is blocked continuously, making the window unresponsive and the progress-bar updates arrive only between frames.

FFmpeg and ffprobe processes are spawned without any lifecycle management. If the user closes the window mid-analysis or if a second analysis is triggered, the child processes keep running. There is no abort path exposed to the renderer.

The IPC progress listener in `preload.js` accumulates unbounded listeners if `onAnalysisProgress` is called more than once without calling `removeProgressListeners`, though the current renderer only calls it once at module load so this is latent rather than active.

<details>
<summary>Issues (5)</summary>

1. **Main-thread blocking** — Frame processing runs synchronously in `stdout.on('data')`, blocking the Electron event loop for the entire analysis duration. Move computation to a worker thread or a child process.
2. **Orphaned FFmpeg on close** — The spawned `ffmpeg` process is never killed when the window closes or analysis is cancelled. Store the `ChildProcess` reference and kill it in `mainWindow.on('closed')` and on re-invocation.
3. **Gamut confidence-mask divergence** — The Python code classifies low-confidence-chromaticity pixels as Rec.709; the JS port falls through to `count2020only`. Apply the same heuristic: treat `sum <= 0` (or very small sum) as Rec.709.
4. **Unbounded buffer growth** — `Buffer.concat` on every `stdout` chunk allocates and copies the entire accumulated buffer each time. Pre-allocate a frame-sized buffer and track a write offset, or use a ring/list approach.
5. **Progress listener leak** — `onAnalysisProgress` adds a new listener on every call without removing prior ones. Either guard against double-registration or call `removeProgressListeners` before re-registering.

## Main-thread compute starvation

The `analyzeVideo` function in `hdr-analyzer.js` hooks `proc.stdout.on('data')` and calls `processFrame` synchronously inside it. `processFrame` iterates over ~2 million pixels (1920×1080 subsampled from 4K) doing three `Math.pow` calls each for PQ EOTF plus matrix multiplication and barycentric tests — roughly 30M floating-point ops per frame. At 1 fps sampling of a 2-hour film that's ~7200 blocking frames.

Because this runs on the main thread, `BrowserWindow` paint, IPC dispatch, and the progress `webContents.send` calls are all deferred until the current `data` handler returns. The user sees a frozen window with the progress bar jumping in bursts between frames. A `worker_threads`-based approach or offloading to a forked process would keep the UI responsive.

## FFmpeg process lifecycle

```js
const proc = spawn(ffmpegPath, args, {
  stdio: ['ignore', 'pipe', 'ignore']
});
```

The `proc` reference is local to the promise closure. Nothing in `main.js` stores it or subscribes to window-close events to kill it. If the user closes the app mid-analysis, FFmpeg continues decoding to a broken pipe and may linger depending on OS pipe semantics. The `start-analysis` handler also has no guard against being called a second time while an analysis is already running — the main-side `lastChartBuffer` would be overwritten by whichever finishes last.

## Gamut classification divergence from Python

The Python version applies a "confidence mask" for pixels with very low XYZ sum:

```python
confidence_mask = s.flatten() > 0.000
valid_in_709[confidence_mask] = in_709[confidence_mask]
# pixels where confidence_mask is False default to True (in_709=True)
```

Pixels with near-zero chromaticity sum are treated as Rec.709 by default. The JS port instead uses:

```js
if (sum > 0.0) {
  // classify normally
} else {
  // Very dark with no color info - treat as 709
  count709++;
}
```

The threshold itself is equivalent (`> 0.0` vs `> 0.000` are the same float comparison), but the behavioral split is different. In the Python code, `valid_in_709` and `valid_in_p3` are initialized to `True` — so pixels that pass the brightness threshold (`>= 1 nit`) but have `sum <= 0` default to Rec.709 without running the barycentric test. The JS code handles that case identically in the `else` branch.

The real divergence is for bright pixels with `X + Y + Z` just barely above zero. The Python code runs the gamut test but then *only applies it* for pixels where `confidence_mask` is True (i.e., `s > 0`). For the remaining pixels, `valid_in_709` stays `True`, defaulting them to Rec.709. The JS code has no such confidence gate — every pixel with `sum > 0.0` gets its barycentric result applied directly. For a pixel with `sum = 0.001`, the division `X/sum` produces extreme chromaticity values that almost certainly fall outside both Rec.709 and P3, classifying the pixel as Rec.2020. This inflates the wide-gamut count for dark HDR content.

Fix: add a minimum-sum threshold (e.g., `sum > 0.01`) below which the pixel defaults to `count709++`.

## Buffer allocation pattern

```js
buffer = Buffer.concat([buffer, chunk]);
```

This allocates a new buffer of size `buffer.length + chunk.length` and copies both on every `data` event. FFmpeg writes in ~64 KB chunks; a single 4K 10-bit frame is ~49.7 MB. That's ~770 concat+copy operations per frame just for the read buffer. A pre-allocated `Buffer.allocUnsafe(frameBytes)` with a write-offset counter would reduce GC pressure substantially.
